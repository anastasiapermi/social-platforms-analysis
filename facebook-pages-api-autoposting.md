# Facebook Pages API — автопостинг

Официальная документация: https://developers.facebook.com/docs/pages-api/posts/
Основные методы: `POST /{page-id}/feed` (текст), `POST /{page-id}/photos` (фото), Video API (видео), `POST /{post-id}` (обновить), `DELETE /{post-id}` (удалить).

## 1. Summary

Facebook Pages API даёт постинг от имени Facebook Page (страницы организации/бизнеса). Поддерживает текст, фото, видео, ссылки. **Главное преимущество: нативное scheduled publishing** через `scheduled_publish_time` (от 10 минут до 30 дней вперёд) — отложенные посты не нужно держать в нашей очереди.

**Сложность интеграции: medium-high** (App Review для prod, привязка к Facebook Login).

## 2. Что нужно для запуска

1. Meta-приложение в https://developers.facebook.com.
2. Facebook Page (организации, бренда, паблика — не личный профиль).
3. App Review для permissions `pages_manage_posts`, `publish_video` (Advanced Access).
4. OAuth flow Facebook Login для получения User access token, затем обмен на Page access token.
5. Пользователь должен иметь роль на Page с tasks: **CREATE_CONTENT, MANAGE, MODERATE**.

## 3. Авторизация

Документация: https://developers.facebook.com/docs/facebook-login/guides/access-tokens

- База API: `https://graph.facebook.com/v25.0`.
- Передача токена: `Authorization: Bearer <PAGE_ACCESS_TOKEN>` или query `access_token=...`.
- Поток:
  1. Facebook Login → User access token.
  2. Обмен на long-lived User token (60 дней).
  3. Запрос Page access token через `GET /me/accounts` — Page tokens **бессрочные**, если получены из long-lived User token.

Page access token принадлежит конкретной странице и используется для действий от её имени.

## 4. Права / scopes

Для автопостинга:

| Scope                       | Назначение                                |
|-----------------------------|-------------------------------------------|
| `pages_manage_posts`        | публикация / обновление / удаление постов |
| `pages_manage_engagement`   | управление взаимодействиями               |
| `pages_read_engagement`     | чтение метрик engagement                  |
| `pages_read_user_content`   | чтение пользовательского контента (комментариев и т.п.) |
| `publish_video`             | публикация видео                          |
| `pages_show_list`           | список страниц пользователя               |

### App Review

Все `pages_*` permissions имеют два уровня доступа:

- **Standard Access** (по умолчанию, без ревью) — permission формально активен, но **работает только для нас и добавленных вручную тестеров** (до 50 человек, либо до 500 с Business Verification). Лимит общий на всё Meta-приложение, а не на платформу: одно приложение покрывает Facebook + Instagram + Threads, и эти 50/500 testers суммарно по всем трём. Подходит только для разработки и демонстрации.
- **Advanced Access** (после прохождения App Review) — permission работает для **всех реальных клиентов**. Только это даёт production-доступ.

То есть **без App Review запустить сервис для клиентов невозможно**.

Срок ревью — от 1-2 недель до нескольких месяцев (1-3 итерации правок — норма). Требования к заявке:
- Скринкаст использования.
- Privacy Policy и Terms of Service на нашем домене.
- Описание use-case.
- Соответствие Platform Policy.

### Page Publishing Authorization (PPA)

Документация: [Get authorized to post or interact as your Page](https://www.facebook.com/help/1939753742723975), [New Authorization for Pages](https://www.facebook.com/business/news/new-authorization-for-pages).

PPA — отдельная процедура Meta по верификации владельца Page. Если Page подпадает под требование, постить на неё (в том числе через `POST /{page-id}/feed` и `POST /{page-id}/photos`) нельзя до прохождения PPA одним из админов.

**Кому именно нужен PPA.** Meta не публикует точный порог в числах. Требование триггерится для Pages «с большим охватом» (high-reach), исторически — это:
- Pages новостных организаций, политических субъектов, общественно-значимых публичных фигур.
- Pages с крупной аудиторией (обычно от десятков-сотен тысяч подписчиков, конкретный порог Meta держит закрытым).
- Pages из категорий, которые Meta помечает как чувствительные к дезинформации.

Если PPA триггерится — у админа в Settings Facebook появится prompt «Complete Page Publishing Authorization», без выполнения которого публикация (в том числе через API) для этой Page блокируется. Через API заранее проверить, нужен ли PPA конкретной Page, **нельзя** — клиент узнаёт только из своего UI.

**Как проходить (со стороны клиента).**
1. В личном Facebook-аккаунте админа: Settings → Identity Confirmation → нажать Get Started в «ID Hub».
2. Загрузить документ, удостоверяющий личность (формат — см. требования в самом UI).
3. Подтвердить страну происхождения и включить 2FA.
4. Meta проверяет от нескольких дней до нескольких недель.
5. После одобрения админ снова может публиковать как Page.

**Что это значит для нашего сервиса.** Через API ситуацию ни предсказать, ни обойти нельзя. В UI подключения Facebook Page имеет смысл превентивно вывести инфоблок: «если у вашей Page крупная аудитория, news- или political-категория — до начала постинга админу нужно пройти Page Publishing Authorization в личных настройках Facebook». При попытке постинга от заблокированного PPA Page API вернёт permission-error — обрабатывать как отдельный тип ошибки в нашем UI и показывать клиенту инструкцию. Отдельно: PPA на Facebook Page **наследуется Instagram-аккаунтом**, привязанным к этой Page (в Facebook Login for Business варианте Instagram API).

## 5. Адресация

Постинг идёт на конкретную Facebook Page по её `page_id`:
```
POST /{page-id}/feed
POST /{page-id}/photos
```

Page access token уже привязан к этой странице — публикация идёт от её имени.

## 6. Постинг текста

Документация: https://developers.facebook.com/docs/pages-api/posts/#publish-posts

- **Лимит длины текста в официальной API-документации не указан.** Поле `message` описано просто как `string`. На стороне нашего сервиса имеет смысл задать собственный валидационный потолок.
- Markdown/HTML не поддерживается, но URL автораспознаётся и формирует сниппет-карточку.
- Можно прикрепить ссылку через параметр `link`.

### Native scheduled publishing

Facebook поддерживает отложенную публикацию **на стороне платформы** через параметр `scheduled_publish_time`: мы отправляем пост с флагом «не публиковать сейчас» и временем публикации, Facebook сам выложит в нужный момент.

- Окно: от **10 минут** до **30 дней** в будущем от момента запроса.
- Время задаётся как Unix timestamp или ISO 8601 (`2026-09-01T10:15:30+01:00`).

Для нашего сервиса это значит: **запланированные посты в окне 10 мин — 30 дней можно отдавать прямо в Facebook**, не держать в нашей очереди. Это снимает с нас часть инфраструктуры на отложенный постинг.

## 7. Постинг медиа

Документация: https://developers.facebook.com/docs/pages-api/posts/#publish-media-posts

- **Фото**: публикуется одним запросом — можно дать URL картинки (Facebook сам скачает) или залить байты через multipart. Возвращается ID поста. **Лимит размера — 10 MB на файл**, для PNG рекомендуется ≤ 1 MB (иначе пикселизация).
- **Видео**: отдельный Video API (https://developers.facebook.com/docs/video-api/). Использует Resumable Upload API (загрузка по чанкам с возможностью докачать после обрыва).
- **Мульти-фото пост**: поддерживается через двушаговый flow — сначала загружаем каждое фото с `published=false` и `temporary=true` (фото хранится в Facebook до 24 часов как unpublished), потом создаём пост с массивом `attached_media: [{media_fbid: ...}, ...]`. Это **полноценный мульти-фото пост в фиде**, не «ссылка на альбом».

### Передача медиа: два варианта инфры

- **Через публичный URL.** Мы передаём Facebook ссылку (`url: https://...`), Facebook анонимно скачивает по ней файл. Простой код. **Требует**: клиентские медиа должны лежать на публично доступном хосте (S3 с public bucket, Cloudflare R2, своя статика). Стоимость CDN-трафика, политика хранения, безопасность (медиа доступно по URL, пусть и непредсказуемому).
- **Через multipart upload (поле `source`).** Шлём байты файла прямо в HTTP-запрос к Facebook. **Не требует** публичного хоста — медиа может лежать где угодно у нас, хоть на локальном диске. Минусы: каждый пост = передача всех байт от нас к Facebook (трафик и время удваиваются — клиент→мы, потом мы→Facebook). Для больших видео нужен отдельный resumable upload — сложнее в реализации.

**Выбор остаётся за решением о приоритетах**: дешевизна разработки vs стоимость инфраструктуры vs характер контента (картинки/видео, маленькие/большие).

## 8. Rate limits

Документация: https://developers.facebook.com/docs/graph-api/overview/rate-limiting/

### Какие токены вообще бывают в Facebook

В Facebook есть три типа токенов, каждый получается своим способом:

| Тип токена | Как получаем | Когда мы его используем |
|---|---|---|
| **App access token** | Формируем сами как `{app-id}\|{app-secret}` из настроек нашего Meta-приложения. Не требует пользователя. | Для общих app-level операций (метаданные, технические запросы). В постинге **не используем**. |
| **User access token** | Получаем, когда клиент проходит Facebook Login OAuth (см. §3). Представляет конкретного пользователя (нашего клиента). Long-lived версия живёт 60 дней. | Только промежуточный шаг — нужен, чтобы получить Page token (через `/me/accounts`). После этого его можно «выбросить». |
| **Page access token** | Получаем из User access token через запрос `/me/accounts`. Представляет конкретную Facebook Page клиента. Бессрочный, если получен из long-lived User token. | **Наш основной токен для постинга.** Под ним идут все вызовы публикации, обновления, чтения по этой Page. |

### Какие лимиты применяются

Facebook применяет разные формулы в зависимости от того, **какой токен** мы используем в запросе:

- **App / User token** → **Platform Rate Limit**: `200 × Number of Users / 1h`
  («Users» = уникальные пользователи нашего приложения за сутки; для нас — это наши платящие клиенты).
- **Page token** → **BUC Rate Limit (Business Use Case)**: `4800 × Engaged Users / 24h` **per-Page**
  («Engaged Users» = люди, взаимодействовавшие с конкретной Page за 24h — лайки, комменты, просмотры).
- Если применимы оба — побеждает BUC.

**Для нашего use-case основной лимит — BUC** (мы постим Page-токеном). Чем популярнее страница клиента, тем больше квота на эту страницу. Для типичного SMM-сценария (2-3 API-вызова на пост, 1-10 постов в день на канал) этот лимит **в сотни раз больше** реальной нагрузки — на практике не блокер.

## 9. Особенности

- **App Review для prod обязателен**, заложить недели.
- **Постить в личные профили нельзя** — только в Pages.
- **Нативный scheduled publishing** — отложенные посты можно класть прямо в Facebook (`scheduled_publish_time`), не держа в нашей очереди. Окно 10 мин — 30 дней.
- **Audience targeting по гео (не для MVP)** — параметр `targeting.geo_locations` позволяет ограничить, кто увидит пост, по странам/городам (например, пост на русском только для подписчиков из РФ/СНГ). Для MVP не нужно; пригодится позже для клиентов с мульти-региональной аудиторией.
- **Формулировка заявки на App Review важна.** В описании use-case при подаче заявки на `pages_manage_posts` нужно представлять наш сервис как «SMM-инструмент для брендов и агентств, который помогает им управлять публикациями на своих Facebook Pages». Расплывчатые формулировки типа «инструмент автоматического постинга» или «AI-публикатор» Meta часто отвергает — это звучит как потенциальный спам-бот. Под «SMM-инструмент для бизнеса» (паттерн Buffer / Hootsuite / Later) Meta привыкла и одобряет охотнее.

## 10. Что учесть при запуске интеграции

- **App Review для `pages_manage_posts` + `publish_video`** — заложить **2-4+ недели** до prod. Параллельно с разработкой готовить скринкаст, Privacy Policy / ToS, Business Verification.
- **UX подключения канала**: клиент проходит Facebook Login → если у него несколько Pages, выбирает одну → мы сохраняем привязку «канал в нашем UI ↔ Page в Facebook».
