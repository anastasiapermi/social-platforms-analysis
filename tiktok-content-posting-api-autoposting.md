# TikTok Content Posting API — автопостинг

Официальная документация: https://developers.tiktok.com/doc/content-posting-api-get-started
Основные методы: `POST /v2/post/publish/creator_info/query/`, `POST /v2/post/publish/video/init/`, `POST /v2/post/publish/content/init/` (фото), `POST /v2/post/publish/status/fetch/`.

## 1. Summary

TikTok Content Posting API позволяет публиковать видео и фото-карусели напрямую в TikTok-аккаунт пользователя. Два режима: **Direct Post** (сразу в профиль) и **Upload** (в TikTok inbox, пользователь сам публикует через приложение). Жёсткие требования: TikTok app approval, **верифицированный домен** для PULL_FROM_URL, **аудит на стороне TikTok** — без него все посты только в режиме `SELF_ONLY` (приватно).

**Сложность интеграции: high.**

## 2. Что нужно для запуска

1. Зарегистрированное приложение на https://developers.tiktok.com.
2. Подключенный продукт **Content Posting API** с конфигурацией **Direct Post**.
3. Approval на scope `video.publish` (и/или `video.upload`).
4. **Верификация домена / URL-префикса** для метода `PULL_FROM_URL`.
5. **Аудит приложения** для снятия ограничения SELF_ONLY на публикуемый контент.
6. OAuth flow (Login Kit) для получения access token и open_id пользователя.

## 3. Авторизация

Документация: https://developers.tiktok.com/doc/login-kit-web

- База API: `https://open.tiktokapis.com/v2/`.
- Upload host: `https://open-upload.tiktokapis.com/`.
- Передача токена: `Authorization: Bearer <ACCESS_TOKEN>`.
- OAuth 2.0 через Login Kit:
  1. Redirect пользователя на TikTok consent page.
  2. Получение `code` → обмен на `access_token` + `open_id`.
  3. `access_token` TTL: **86 400 секунд (24 часа)**.
  4. `refresh_token` TTL: 365 дней.

`open_id` — это идентификатор пользователя в контексте приложения (анонимизированный, не TikTok user_id).

## 4. Права / scopes

| Scope            | Назначение                                              |
|------------------|---------------------------------------------------------|
| `video.publish`  | прямая публикация в профиль пользователя (Direct Post)  |
| `video.upload`   | загрузка в Inbox пользователя (он публикует сам)        |
| `user.info.basic`| базовая информация (имя, аватар)                        |

### App Audit / Approval

- Каждый scope требует approval при создании приложения.
- **Аудит контента**: до прохождения аудита TikTok все посты, опубликованные через API, **принудительно идут в `SELF_ONLY`** (видны только автору). Чтобы снять это ограничение — пройти review:
  - Tested integration.
  - Compliance with Terms of Service.
  - Privacy Policy / data handling.

Срок аудита — недели.

### Domain verification

Для `source=PULL_FROM_URL` домен (или URL-префикс), с которого TikTok тянет медиа, должен быть **верифицирован** в Developer Portal (через HTML meta tag или DNS-запись).

## 5. Адресация

Посты идут в аккаунт пользователя, который авторизовал приложение. Целевой `open_id` зашит в access token.

Два режима постинга — детали в §7:
- **Direct Post** (`video.publish`) — наш API публикует сразу в профиль. Требует аудит.
- **Upload (Inbox)** (`video.upload`) — наш API кладёт видео в drafts, пользователь сам публикует через TikTok-приложение. Не требует аудит.

## 6. Постинг текста

Чисто-текстовые посты **не поддерживаются**. TikTok — медиа-платформа: всё крутится вокруг видео и фото-каруселей. Текст живёт в:
- `title` (для видео-поста) — до **2200 UTF-16 runes** (подтверждено в [Direct Post reference](https://developers.tiktok.com/doc/content-posting-api-reference-direct-post)).
- `title` (для фото-поста) — до **90 UTF-16 runes**.
- `description` (для фото-поста) — до **4000 UTF-16 runes** (подтверждено в [Photo Post reference](https://developers.tiktok.com/doc/content-posting-api-reference-photo-post)).

Хэштеги (`#`) и упоминания (`@`) распознаются автоматически (разделители — пробелы и новые строки).

## 7. Постинг медиа

Документация: [Content Posting API](https://developers.tiktok.com/doc/content-posting-api-get-started), [Direct Post](https://developers.tiktok.com/doc/content-posting-api-reference-direct-post/), [Upload (Inbox)](https://developers.tiktok.com/doc/content-posting-api-reference-upload-video/).

TikTok даёт два независимых режима со своими endpoints, scopes и требованиями. Выбор между ними — стратегический вопрос проекта.

### Сравнение режимов

| Аспект | Direct Post | Upload (Inbox) |
|---|---|---|
| Endpoint | `/v2/post/publish/video/init/` | `/v2/post/publish/inbox/video/init/` |
| Scope | `video.publish` | `video.upload` |
| Кто публикует | Наш API | **Пользователь** в TikTok-приложении |
| Требует аудит для public постов | **Да** (иначе SELF_ONLY) | **Нет** — обходит SELF_ONLY |
| Аккаунт обязан быть приватным (без аудита) | Да | Нет |
| Подпись/хэштеги/privacy задаются | В нашем API | **В TikTok-приложении** пользователем |
| Полностью автоматический постинг | Да | Нет — пользователь делает финальный шаг |
| Подходит для расписания | Да | Зависит от пользователя |

### Direct Post — flow

1. **`creator_info/query`** — узнать настройки creator'а (privacy options, max duration, состояние duet/comment/stitch). UX-обязательство TikTok: показать пользователю avatar, name и privacy options перед публикацией.
2. **`video/init`** (или `content/init` для фото) — отправить `post_info` (title, privacy_level, disable_duet/comment/stitch, обложка) и `source_info`. Два варианта источника: `FILE_UPLOAD` (мы шлём байты) или `PULL_FROM_URL` (TikTok скачивает с нашего **верифицированного** домена).
3. Для `FILE_UPLOAD` — PUT байтов на полученный `upload_url`, поддерживается chunked upload (resumable, каждый chunk с `Content-Range`).
4. **`status/fetch`** — polling статуса по `publish_id`.

Для фото (`content/init`) обязательны параметры `post_mode=DIRECT_POST` и `media_type=PHOTO`, `photo_images[]` — массив URL с верифицированного домена, `photo_cover_index` — индекс обложки, опционально `auto_add_music: true`.

### Upload (Inbox) — flow

Этот режим обходит главное ограничение неаудированных клиентов (SELF_ONLY) ценой UX: вместо прямой публикации мы кладём видео в TikTok-инбокс пользователя, и он сам завершает публикацию в приложении.

1. **`inbox/video/init`** — инициализация upload в инбокс. `post_info` **не передаётся** (пользователь введёт сам), только `source_info`. Scope `video.upload`.
2. PUT байтов на полученный `upload_url` (chunked).
3. TikTok отправляет пользователю push-уведомление; пользователь открывает TikTok-приложение, находит видео в drafts/inbox, заполняет подпись/хэштеги/privacy и тапает «Post».

Прямая цитата из документации TikTok: *«You should inform users that they must click on inbox notifications to continue the editing flow in TikTok and complete the post»*.

### Когда Upload mode имеет смысл

- **MVP до прохождения аудита** — можем заявить TikTok-интеграцию без ожидания месяцев на review.
- **Compliance-сценарии** — где бренд требует ручное подтверждение публикации в нативном приложении.
- **«Share to TikTok» паттерн** — как кнопка «Поделиться» в видеоредакторах.

### Что мы теряем в Upload mode

- **Не полный автопостинг** — пользователь должен делать действия в TikTok app для каждого поста.
- **Подпись / хэштеги в нашем UI не передаются в TikTok** — пользователь заполнит их заново в TikTok-приложении.
- **Scheduled publishing ломается** — публикация зависит от того, открыл ли пользователь TikTok вовремя.
- **«Черновик → ревью → публикация»** превращается в **«черновик → ревью у нас → ещё один шаг в TikTok»** — лишняя ступень.

### Стратегия выбора для нашего проекта

- **TikTok критичен в MVP** → подаваться на аудит параллельно с разработкой, использовать **Direct Post** с момента запуска.
- **TikTok вторичен** → запустить MVP на **Upload mode** (без ожидания аудита, но с пометкой пользователю «финальная публикация в TikTok-приложении»). После прохождения аудита через несколько месяцев — переключиться на Direct Post.
- **TikTok необязателен** → исключить из MVP.

### Спецификации медиа

**Video:** MP4 (рекомендуется H.264 + AAC), длительность зависит от аккаунта, точное значение приходит в `max_video_post_duration_sec` от `creator_info` (типичный example в документации — **300 сек / 5 минут**). Aspect ratio 9:16 рекомендуется.

**Photo:** WEBP (рекомендуется), JPG, PNG; **до 35 изображений** в карусели (подтверждено: *«An array containing up to 35 photo content URLs»*); все из верифицированного домена.

## 8. Rate limits

### Per-request лимит

- **6 запросов / минуту** на каждый access_token (подтверждено в Content Posting API reference). При превышении — HTTP 429 + `rate_limit_exceeded`.

### 24h active creator cap (лимит на наш сервис целиком)

«Active creator» — TikTok-аккаунт, который **за последние 24 часа** хотя бы раз опубликовал пост через наш API. «Cap» — потолок на их количество в сутки.

**Кто устанавливает:** TikTok — на основе цифр, которые **мы сами заявляем в форме аудита**. В заявке есть поле «сколько активных пользователей планируете обслуживать в день». Если мы пишем «1000» — TikTok ставит cap около 1000.

**На что распространяется:** на **наше приложение целиком** (per API client), не на каждого отдельного клиента.

**Пример:**
- Мы заявили в аудите: 1000 active creators / day.
- В пятницу к 11:00 UTC 1000 наших клиентов уже сделали хотя бы один пост через наш сервис.
- 1001-й клиент в 11:01 пытается опубликовать → TikTok возвращает `reached_active_user_cap`. До полуночи UTC этот клиент не сможет постить через наш сервис.

**Что с этим делать:** при росте подавать **повторную заявку** в TikTok за увеличением cap'а. Это административный шаг, не отказ от сервиса. Стратегически — **заявлять цифры с запасом** в первой форме аудита, чтобы не упереться в потолок в первый же месяц после запуска.

### Posting cap (лимит на каждого нашего клиента)

Один TikTok-аккаунт (наш клиент) может опубликовать через **любые API** (не только наш) **~15 постов в сутки** в режиме Direct Post.

**Кто устанавливает:** TikTok зашил жёстко в платформу. Не зависит от нашего аудита, не настраивается.

**Шарится между всеми API-клиентами:** если наш клиент использует одновременно нас + Hootsuite + Buffer, **их посты через API складываются** в общий счётчик 15/день. Если в Hootsuite клиент уже опубликовал 10 постов, у нас он сможет опубликовать максимум ещё 5.

**Пример:**
- Бренд «Кофейня X» подключил TikTok к нам и параллельно к Buffer.
- В 9:00 они через Buffer выложили 12 постов.
- В 14:00 хотят выложить через нас ещё 5 — успешно публикуются только 3, потом TikTok вернёт лимит.

**Что с этим делать:**
- На практике для большинства брендов **не блокер** — 15 постов/день в TikTok это много, нормальный темп = 1-3 поста/день.
- В UI хорошо бы показывать клиенту **счётчик «осталось N постов до конца суток»**, особенно если он использует другие SMM-инструменты параллельно. Иначе он получит непонятную ошибку.

### Сравнение двух лимитов

| Аспект | 24h active creator cap | Posting cap |
|---|---|---|
| Уровень | На наше приложение целиком | На каждый TikTok-аккаунт клиента |
| Кто устанавливает | TikTok по нашей форме аудита | TikTok жёстко, ~15/день |
| Реальный блокер? | Да, при росте — требует повторной заявки | Только для очень активных брендов |
| Меняется ли | Да, через увеличение cap'а | Нет, фиксирован |
| Ошибка API | `reached_active_user_cap` | стандартный rate-limit response |

### Прочее

- ~**1 000 calls / день** на приложение для creator-info (запросы информации о creator перед публикацией).

## 9. Особенности

- **Аудит контента — главный блокер**. До его прохождения все публикации идут в `SELF_ONLY` независимо от `privacy_level` параметра. Сторонним приложениям без аудита фактически нельзя постить публично.
- **Верификация домена для PULL_FROM_URL** — без неё этот режим не работает. Альтернатива — FILE_UPLOAD с PUT-байтами, не требует верификации (но медиа должно быть локально/на нашем сервере).
- **Текстовых постов нет** — только видео и фото-карусели.
- **Лимиты текста**: video `title` до 2200 UTF-16 runes, photo `title` до 90, photo `description` до 4000 — задокументировано в API reference.
- **Native scheduled publishing отсутствует** — отложенное у нас.
- **Upload mode (Inbox)** — альтернатива Direct Post: контент попадает в drafts, пользователь публикует руками в TikTok app. Снимает часть требований к аудиту, но UX хуже.

## 10. Что учесть при запуске интеграции

- **Content Audit на стороне TikTok** — обязательно до запуска публичных постов в production. Заложить **недели**. Без аудита все посты идут в `SELF_ONLY` и аккаунт клиента должен быть приватным — для SMM-сценария неприменимо.
- **Верификация домена** в TikTok Developer Portal — если используем `PULL_FROM_URL` для медиа.
- **UX подключения канала**: клиент проходит OAuth через TikTok Login Kit → выдаёт нам permissions (`video.publish`). Access token 24h — нужен ежедневный рефреш через refresh-token (365 дней).
- **UX публикации**: перед каждой публикацией показывать клиенту его TikTok creator info (avatar/nickname/privacy options) — это обязательное требование UX-guideline TikTok при аудите.
