Сводка по всем исследуемым площадкам: тип авторизации, сложность интеграции, требования ревью, бюрократия, риски. Все факты подтверждены по живой документации.

> **Facebook / Instagram / Threads — одна строка, потому что общая бюрократия**: одно Meta-приложение, одна Business Verification, один комплект Privacy Policy / Terms, один скринкаст. Но **Permissions Review проходит отдельно по каждому праву** (`pages_manage_posts`, `instagram_content_publish`, `threads_content_publish`) — Meta может одобрить одно и отказать в другом. Срок и риск общие, потому что заявки идут параллельно одним пакетом.

> **Что такое App Review в принципе.** Это процедура, через которую платформа проверяет, что наше приложение действительно делает то, что заявлено, и не злоупотребляет API. До прохождения этого гейта приложение либо работает только в режиме разработки (для тестовых пользователей), либо постит только приватно (TikTok), либо имеет жёсткие лимиты. Это **наш** проектный риск: код можно написать за дни, а ревью занимает недели и может закончиться отказом.

## 1. Telegram

Подробный документ: [telegram-bot-api-autoposting.md](./telegram-bot-api-autoposting.md)

**Ревью: нет.**

Бот создаётся в `@BotFather` командой `/newbot`, токен выдаётся сразу. Никакой проверки приложения, никаких документов.

**Архитектура: каждый клиент создаёт свой бот.** Это значит, что для подключения Telegram-канала клиент должен сам:
1. Зайти в `@BotFather`, командой `/newbot` создать своего бота и получить токен.
2. **Передать нам этот токен** (вставить в наш UI). Без него мы не можем постить от имени его бота.
3. Добавить своего бота админом в свой канал с правом `can_post_messages`.

**Что от нас:** код + подробная пошаговая инструкция в UI «как создать бот, скопировать токен, передать его нам и добавить бот в канал админом».

## 3. X / Twitter

Подробный документ: [x-twitter-api-autoposting.md](./x-twitter-api-autoposting.md)

Документация: https://docs.x.com/x-api/getting-started/getting-access и https://docs.x.com/x-api/getting-started/pricing

**Ревью: НЕТ.** X отказался от tier'ов в пользу pay-per-usage с покупкой кредитов.

### Что делаем мы один раз

1. Зайти на https://console.x.com, войти через наш корпоративный X-аккаунт.
2. Принять Developer Agreement.
3. Создать наше app, получить `client_id` / `client_secret`, настроить `redirect_uri` под наш сервис.
4. Запросить scopes: `tweet.write`, `media.write`, `offline.access` (последний — чтобы refresh-токен выдавался).
5. **Купить кредиты** в Developer Console. Списываются с нашего корпоративного баланса при каждом API-вызове.

### Как клиент подключает свой X к нашему сервису

Клиент проходит стандартный **OAuth 2.0 с PKCE** (Authorization Code Flow with Proof Key for Code Exchange):

1. В нашем UI клиент жмёт «Подключить X».
2. Мы редиректим его на `https://twitter.com/i/oauth2/authorize?...` с нашими client_id и нужными scopes.
3. Клиент логинится в свой X (на стороне X — мы пароль не видим), подтверждает: «приложение [наше имя] хочет публиковать от моего имени».
4. X редиректит обратно на наш `redirect_uri` с `code`.
5. Наш backend обменивает `code` на `access_token` (короткий TTL) + `refresh_token` (используем для рефреша). Сохраняем эту пару в нашей БД, привязываем к клиенту.
6. С этого момента мы можем постить от его имени через `POST /2/tweets` под его токеном. Кредиты списываются с **нашего** баланса в Developer Console — клиент за API-вызовы напрямую не платит, он платит нам по нашему тарифу.

Документация на flow: https://docs.x.com/fundamentals/authentication/oauth-2-0/user-access-token.

### Цены (на момент проверки)

| Операция | Цена |
|---|---|
| Content: Create (пост) | **$0.015 / запрос** |
| Content: Create with URL | **$0.200 / запрос** |
| Content: Manage | $0.005 / запрос |
| Posts: Read | $0.005 / ресурс |
| User: Read | $0.010 / ресурс |

Полный pricelist: https://docs.x.com/x-api/getting-started/pricing.

X — единственная платформа, где **мы платим за каждый пост наших клиентов из своего кармана** (или закладываем в цену тарифа).

### Срок и риск

- Срок «получения доступа»: моментально после оплаты.
- Риск: **нет административного риска**, только финансовый. Внезапные изменения цен — отдельная угроза: X в феврале 2026 уже сменил модель — отменил фиксированные tier'ы (Basic / Pro / Enterprise по подписке) и перевёл всех на pay-per-use с покупкой кредитов. Официальный анонс: [Announcing the Launch of X API Pay-Per-Use Pricing](https://devcommunity.x.com/t/announcing-the-launch-of-x-api-pay-per-use-pricing/256476).

## 4. OK (Одноклассники)

Подробный документ: [odnoklassniki-api-autoposting.md](./odnoklassniki-api-autoposting.md)

Документация: https://apiok.ru/dev/app/create

**Ревью: модерация приложения** в кабинете devaccess.

### Процесс

«Devапп» здесь — это **запись в Developer Console OK** (не наш сайт). Через неё OK выдаёт нам `application_key` и `application_secret_key` для вызовов API. Тот же паттерн, что Meta App в developers.facebook.com или X Developer App в console.x.com.

1. Зарегистрироваться на ok.ru, привязать email.
2. Получить права разработчика на https://ok.ru/devaccess.
3. Создать devапп: название и описание (≤250 символов) — то, что увидит клиент при OAuth-консенте; уникальное имя в ссылке.
4. **Подключить платформу OAuth** к devаппу (нужна для автопостинга в группу/ленту).
5. Указать `redirect_uri` для OAuth — URL на нашей стороне, куда OK вернёт клиента после approve.
6. Запросить permissions: `VALUABLE_ACCESS`, `GROUP_CONTENT`, `PHOTO_CONTENT`, `VIDEO_CONTENT`.
7. Ключи devаппа приходят на email.

### Срок и риск

- Срок: дни (быстрее всех остальных платформ).
- Риск отказа: **низкий** для обычных бизнес-сценариев.

## 5. YouTube (API Services Compliance Audit)

Подробный документ: [youtube-data-api-autoposting.md](./youtube-data-api-autoposting.md)

Документация: https://developers.google.com/youtube/v3/guides/quota_and_compliance_audits

**Ревью: НЕ обязательно** для базового использования с квотой по умолчанию. **Обязательно** для повышения квоты сверх 10 000 units/день.

### Что значит «default»

- Квота 10 000 units/день per project.
- `videos.insert` = **100 units** → максимум **100 загрузок видео/день** на наш проект.
- При нашем мульти-клиентском use-case это **жёсткое ограничение**.

### Когда нужен Audit

Если планируем больше 100 загрузок/день суммарно — нужно проходить **Compliance Audit**. Это:
- Подача формы **YouTube API Services - Audit and Quota Extension Form**.
- Описание use-case, scopes, целевой аудитории.
- Соответствие YouTube API Services Terms of Service.
- Privacy Policy / data handling.

### Periodic Audits

YouTube периодически переаудитит приложения с большой квотой. Может потребовать переоформление документов.

### Срок и риск

- Срок: от нескольких недель до нескольких месяцев.
- Риск: средний. При отказе можно (а) подать **апелляцию через [Appeals Form](https://developers.google.com/youtube/v3/guides/quota_and_compliance_audits)**, (б) исправить замечания и переподаться, (в) остаться на default-квоте 10 000 units/день = 100 загрузок/день — это не блокер для MVP, упрёмся в неё только при значительном росте.

## 6. Facebook / Instagram / Threads (Meta App Review)

Подробные документы: [facebook-pages-api-autoposting.md](./facebook-pages-api-autoposting.md) · [instagram-graph-api-autoposting.md](./instagram-graph-api-autoposting.md) · [threads-api-autoposting.md](./threads-api-autoposting.md)

Документация: https://developers.facebook.com/docs/resp-plat-initiatives/individual-processes/app-review

**Ревью: ОБЯЗАТЕЛЬНО.** Без него приложение в Development Mode — работает только для нас + добавленных вручную в Dashboard тестеров (**до 50 реальных людей** в стандартной конфигурации, либо **до 500** при подключённой Business Verification — см. https://developers.facebook.com/docs/development/build-and-test/app-roles, раздел «Limitations»). **Важно: 50 / 500 — это общий пул на всё Meta-приложение, а не на каждую платформу отдельно.** Одно Meta-приложение покрывает Facebook + Instagram + Threads, и testers считаются на приложение целиком: один tester в нашем приложении автоматически может тестировать FB, IG и Threads — это один слот, а не три.

### Из чего состоит App Review

1. **App Verification** — Meta-ревьюеры тестируют наше приложение, проверяя что оно реально работает и use-case соответствует заявленному. Если ревьюер не может протестировать функцию (например, из-за неработающего демо) — отказ.
2. **Permissions Review** — проверка, что мы запрашиваем **только** необходимые scopes (для нас: `pages_manage_posts`, `publish_video`, `instagram_content_publish`, `threads_content_publish`).
3. **Business Verification** (с 12 июня 2023 для advanced permissions) — верификация юрлица: документы компании, BIN, регистрация. **Это отдельный процесс** в настройках Business Manager.
4. **Data Handling Questions** — анкета: как мы храним данные пользователей, как защищаем, кто имеет доступ.
5. **Заполнение профиля приложения** в кабинете Meta для разработчиков (раздел «App Settings» в Developer Console). Перед отправкой на ревью нужно заполнить базовые поля: иконка приложения, ссылка на нашу **Privacy Policy**, **категория** приложения из списка Meta, **business email** для связи, перечень **платформ**, где работает приложение (web/iOS/Android). Без заполнения этих полей Meta не принимает заявку на ревью.

### Что подготовить заранее

- **Скринкаст** с прохождением полного UX-flow: пользователь подключает Facebook → пишет пост → ревьюит → публикует → видит результат на Facebook Page. Скринкаст должен быть исчерпывающим, без сокращений.
- **Privacy Policy** и **Terms of Service** на наш домен. Реальные, юридически грамотные документы — Meta их читает.
- **Описание use-case** письменно: почему нам нужен `pages_manage_posts`, какой проблемой это решается, как пользователь от этого выигрывает.
- **Готовый Business-аккаунт** с верифицированной компанией. Если у нас нет — заложить отдельные недели на business verification.
- **Работающее тестовое приложение** в Development Mode на нашем стенде, доступное Meta-ревьюеру.

### Срок и риск

- Срок ревью: от 1-2 недель до нескольких месяцев. Meta часто отправляет на доработку с замечаниями (1-3 итерации — норма).
- **Риск отказа для нашего профиля (SMM-продукт со средним трекшном): низкий-средний.** Meta предпочитает понятные B2B-кейсы (брендинг, агентства), и наш use-case попадает прямо в этот сегмент: «помогаем брендам публиковать на собственных страницах через единый интерфейс с черновиком и ревью». Главный риск — не «откажут совсем», а «вернут на доработку 1-2 раза» (некачественный скринкаст, недостаточная Privacy Policy, проблемы Business Verification).
- Одно Meta-приложение можно использовать для FB+IG+Threads сразу — ревью идёт параллельно по каждому из них в одном пакете.

### Альтернативы

- **Standard Access без ревью** — работает, но только для добавленных вручную тестеров (до 50 без Business Verification, до 500 с BV — см. https://developers.facebook.com/docs/development/build-and-test/app-roles). **Лимит общий на всё Meta-приложение, а не на платформу**: 50/500 testers суммарно по Facebook + Instagram + Threads, один tester занимает один слот на все три. Не для production.

## 7. LinkedIn (Marketing Developer Platform)

Подробный документ: [linkedin-posts-api-autoposting.md](./linkedin-posts-api-autoposting.md)

Документация: https://learn.microsoft.com/en-us/linkedin/marketing/increasing-access

**Ревью: ОБЯЗАТЕЛЬНО для company-постов (`w_organization_social`).**

### Доступ к LinkedIn API проходит в две оси

**Ось 1: в какую программу нас примут**

LinkedIn делит API на программы (пакеты эндпоинтов + permissions). Программы бывают двух типов:
- **GA (Generally Available)** — открыта всем, кто подаёт заявку. LinkedIn оценивает по формальным критериям, большинство адекватных кейсов одобряет. Срок: неделя-две.
- **Restricted** — закрытая, LinkedIn сам выбирает партнёров. До 60 дней ответ, могут отказать без объяснения.

Все наши нужные scopes (`w_member_social`, `w_organization_social`) идут через **Community Management API**, которая является **GA-программой**. То есть подаём стандартную заявку — большая вероятность одобрения.

**Ось 2: какой уровень доступа внутри программы**

После одобрения в программу мы попадаем не сразу на production. У Community Management API два уровня:

| Tier | Возможности | Ограничения |
|---|---|---|
| **Development** | Полный API, для разработки/тестирования | Урезанные rate limits (sandbox-объёмы); **должны завершить интеграцию в течение 12 месяцев** |
| **Standard** | Production, без ограничений | Отдельная заявка после Development tier |

Все приложения **начинают с Development tier**. Чтобы получить Standard — подаём **вторую заявку** после того, как интеграция реально работает с тестовыми клиентами.

### Полный путь для нас

```
[Подаём заявку в Community Management API (GA)]
            ↓ ~1-2 недели
[Одобряют → Development tier]
            ↓ строим интеграцию, подключаем первых клиентов
[Подаём вторую заявку — на Standard tier]
            ↓ показываем работающую интеграцию
[Standard tier → production без ограничений]
```

Главное: 12-месячное окно Development — жёсткий дедлайн. За год нужно дотащить интеграцию и перейти на Standard, иначе доступ могут отозвать.

### Что подготовить

- Описание use-case на нашем сайте.
- Privacy Policy и Terms of Service.
- Заявка через https://www.linkedin.com/developers/apps → «Products» → запрос Community Management API.

### Срок и риск

- Срок: **не зафиксирован публично**, для GA-программ обычно неделя-две на принятие в Development. Standard tier — отдельный цикл.
- Риск: **средний**, для понятных бизнес-сценариев одобряют. Для нишевых use-кейсов могут отказать.

## 8. VK (email-одобрение scope)

Подробный документ: [vk-api-autoposting.md](./vk-api-autoposting.md)

Документация: https://dev.vk.com/ru/api/access-token/user-token

**Ревью: НЕ формальное, но scope `wall` и `photos` для user-токена** выдаются «в исключительных случаях через запрос в поддержку».

### Прямая цитата

> Для запроса таких прав напишите на `devsupport@corp.vk.com`.

### Что писать

Описание use-case, обоснование, объём планируемых вызовов, реквизиты юрлица. Ответ — через переписку с поддержкой VK.

### Обход через community-токен

**Community access token не имеет этого ограничения.** Если бизнес-сценарий — «постить в сообщества пользователей» (а не «в личную ленту»), community-токен решает проблему: админ сообщества создаёт ключ в настройках своей группы с нужными правами сам, без VK support.

**Для нашего продукта:** наши клиенты — это владельцы/админы VK-сообществ, которые хотят постить туда через наш UI. Community-токен подходит идеально: владелец сообщества сам создаёт ключ в настройках своей группы и передаёт нам — никакой переписки с VK support, никакой зависимости от их одобрения. User-токен (постинг от имени конкретного человека) для нас менее релевантен и требует email-апрува VK без гарантии.

### Срок и риск

- Срок переписки: дни-недели.
- Риск отказа для user-токена: **высокий** (формулировка «в исключительных случаях»).
- Через community-токен: **рисков нет**, всё в руках админа сообщества.

## 9. TikTok (Content Posting Audit)

**Аудит: ОБЯЗАТЕЛЬНО для публичного постинга.** Без аудита все посты идут в режиме `SELF_ONLY` (приватно, виден только автору).

### Ограничения для неаудированных клиентов

Документация: https://developers.tiktok.com/doc/content-sharing-guidelines

- **5 пользователей** максимум в окне 24h.
- Все аккаунты этих пользователей **должны быть приватными** (глобальная настройка профиля «Private account») **на момент публикации**. Иначе API возвращает ошибку `unaudited_client_can_only_post_to_private_accounts`.
- Все посты идут только в **SELF_ONLY** viewership (видит только автор).
- Чтобы пост стал публичным после — пользователь должен **вручную в TikTok-приложении**: (1) переключить аккаунт обратно в public, (2) для **каждого поста отдельно** поменять видимость с «Only me» на «Everyone». Batch-API для этого нет.

**Что это значит для нас:** наши клиенты — бренды и креаторы, которые подключают свои TikTok-аккаунты к нашему сервису и платят за автопостинг. Им нужны **публичные посты** — иначе зачем они вообще покупают SMM-инструмент. Без аудита **наш сервис не сможет дать им это**: их аккаунты пришлось бы держать приватными, и каждый пост они вручную «допроявляли» бы в TikTok app. Это не продаваемая фича. **Аудит — обязательный гейт до того, как мы откроем TikTok-интеграцию клиентам в production.**

### Глобальные лимиты (даже после аудита)

- **24h active creator cap** — на основе данных из формы аудита (мы заявляем планируемый объём).
- **Posting cap**: ~15 постов в день на одного creator-аккаунта, шарится между всеми API-клиентами.

### Политика «authentic creators» (риск отказа)

TikTok прямо запрещает:
- **Cross-posting** контента из других платформ (если TikTok сочтёт, что мы перепаковываем чужой контент — отказ).
- **Internal/private use** — приложение должно быть для широкой аудитории, не для собственной команды/одного клиента.
- **Watermarks** и брендинг чужих платформ в самом видео — нарушение правил, может привести к удалению контента.

### Что подготовить

- Описание use-case (заявка через Developer Portal).
- **Демонстрация UX**: TikTok смотрит, как именно creator видит и публикует пост в нашем UI.
- **Compliance с UX guidelines**:
  - Показывать creator info (avatar, username, nickname) перед публикацией.
  - Privacy status — dropdown без default значения; опции только из `privacy_level_options` от `creator_info` API.
  - Длительность видео ≤ `max_video_post_duration_sec` из `creator_info`.
- Verified домен (через HTML meta tag или DNS) — для `PULL_FROM_URL` flow.

### Срок и риск

- Срок: недели.
- Риск: **высокий**, особенно если наш use-case похож на «cross-posting контента из других соцсетей». Нужно чётко позиционировать как «authentic creators делают оригинальный контент».

### Альтернатива аудиту: Upload (Inbox) режим

TikTok даёт **два независимых режима** Content Posting API. Они различаются принципиально по тому, кто публикует пост — мы или пользователь — и поэтому имеют разные требования к аудиту.

| Аспект | Direct Post | **Upload (Inbox)** |
|---|---|---|
| Endpoint | `/v2/post/publish/video/init/` | `/v2/post/publish/inbox/video/init/` |
| Scope | `video.publish` | `video.upload` |
| Кто публикует | Наш API сразу в профиль | **Пользователь** в TikTok-приложении |
| Аудит для public постов | Обязателен | **Не нужен** |
| Аккаунт обязан быть приватным | Да (без аудита) | Нет |
| Финальная публикация | Автоматическая | Ручная в TikTok app |

**Как работает Upload mode:**

1. Наш API заливает видео в инбокс/drafts пользователя через `/v2/post/publish/inbox/video/init/`.
2. TikTok автоматически шлёт пользователю push-уведомление «У тебя новое видео».
3. Пользователь открывает TikTok-приложение, видит видео в drafts.
4. **Внутри TikTok** пользователь сам заполняет подпись, хэштеги, выбирает privacy (Public / Friends / Only me).
5. Жмёт «Post» — публикация.

Прямая цитата из TikTok docs:

> You should inform users that they must click on inbox notifications to continue the editing flow in TikTok and complete the post.

**Почему обходит аудит:** TikTok боится автоматизированного спама от Direct Post. В Upload mode финальное решение за человеком, спам-риск ниже → проверок меньше.

**Что мы теряем:**

- Подпись/хэштеги из нашего UI **не переносятся в TikTok** — пользователь заполнит их заново.
- Это **не полный автопостинг** — каждый пост требует действия пользователя в TikTok-приложении.
- **Расписание ломается** — публикация зависит от того, когда пользователь откроет TikTok.
- **«Черновик → ревью у нас → ещё один шаг в TikTok»** — лишняя ступень в UX.

**Стратегия выбора:**

- **TikTok критичен в MVP** → подаём на аудит параллельно с разработкой, **Direct Post** с момента запуска.
- **TikTok вторичен** → запускаем MVP на **Upload mode** (без ожидания аудита), честно сообщая пользователю «финальная публикация в TikTok-приложении». Через несколько месяцев после прохождения аудита — переключаемся на Direct Post.
- **TikTok необязателен** → исключаем из MVP.

## Что это значит для нашего проекта

### Платформы, готовые к интеграции сразу

- **Telegram, MAX** — добавляем в MVP без задержек.

### Платформы с быстрым approval (недели)

- **OK** — модерация в Developer Portal, обычно дни.
- **VK** через community-токен — без ревью, под контролем админа сообщества.
- **X** — мгновенно после покупки кредитов.

### Платформы со средним gate (1-3 месяца)

- **LinkedIn** — Community Management API, Development tier.
- **YouTube** — если не упираемся в 100 загрузок/день, можно вообще без аудита; иначе Compliance Audit.

### Платформы с тяжёлым gate (месяцы, риск отказа)

- **Meta (FB / Instagram / Threads)** — App Review + Business Verification. Самый дорогой проектный риск.
- **TikTok** — content audit + жёсткие политики против cross-posting.

### Рекомендация по последовательности

1. **MVP с Telegram + MAX + VK (community)** — запускается за недели разработки без внешних approval.
2. **Параллельно**: подаём заявку на Meta App Review (FB+IG+Threads одним пакетом). Готовим скринкаст, Privacy Policy, Business Verification. Заложить 2-3 месяца.
3. **После MVP**: LinkedIn, OK, YouTube, X. Каждый — отдельный спринт после прохождения Meta.
4. **TikTok — отдельно**, как наиболее рисковый. Возможно, начать с режима Inbox/Upload (пользователь сам публикует через TikTok app) — это снимает часть требований аудита.
